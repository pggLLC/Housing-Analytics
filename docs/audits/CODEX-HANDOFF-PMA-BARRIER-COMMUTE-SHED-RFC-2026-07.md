# Codex Handoff — PMA Barrier + Commute-Shed RFC

**For:** Claude QA / scoping first, then Codex implementation only after owner sign-off
**Issue:** #1232
**Created:** 2026-07-17
**Status:** RFC handoff only — do not write production code from this doc alone

## Why This Exists

Issue #1232 had three parts:

1. **PR A — display honesty:** render the PMA as included tract polygons instead
   of a convex hull. Done in #1237.
2. **PR B — analytic apportionment:** replace bbox/distance proxy weights with
   tract-polygon area shares inside the circular buffer. Done in #1238.
3. **RFC — barrier-aware and commute-shed delineation:** still open. This is
   the remaining work, and it needs a separate scoped decision record before
   code.

The next agent must not jump straight into implementation. The remaining phase
can change which tracts are included, not just how existing included tracts are
displayed or weighted. That means it can move PMA scores materially and needs a
methodology decision, calibration evidence, migration disclosure, and explicit
owner approval.

## Current Verified Baseline

- `js/pma-delineation.js` renders included tract polygons from
  `data/market/pma_tract_display_geometry.geojson`.
- `js/market-analysis.js` computes tract `_bufferShare` using tract polygon
  share inside the selected circular buffer, with bbox fallback when geometry is
  unavailable.
- `docs/PMA_SCORING.md` documents the 2026-07-17 migration from bbox shares to
  polygon apportionment and representative score movement.
- `data/market/natural_barriers_co.geojson` exists and is already covered by
  the Data Map coverage guard. It contains highway and water barrier features
  generated by `scripts/market/fetch_natural_barriers.py`.
- `js/pma-barriers.js` exists but is not implementation-ready. It contains
  older heuristic exclusion factors and a straight-line site-to-centroid
  crossing helper. Treat it as prior art to audit, not as an approved method.
- `js/pma-commuting.js` exists but is not implementation-ready for this phase.
  It can fall back to synthetic workplaces and convex-hull boundaries; those
  behaviors are unacceptable for shipped PMA delineation.
- `js/data-connectors/lodes-commute.js` and `data/market/lodes_co.json` provide
  the current LODES data surface. Any commute-shed proposal must use real
  committed data or explicitly stay null/disabled.

## Owner Decisions Already Recorded

From the #1232 owner clarification:

- **No geometric/decorative clipping.** Do not cut tract polygons at highways,
  rivers, rail lines, or the buffer arc for visual effect.
- **Partial tracts render as whole tracts with weight-proportional opacity.**
  An arc slice through a neighborhood implies sub-tract precision that the data
  does not support.
- **The dashed buffer ring remains the selection-domain overlay.**
- **Barrier cut-offs only appear when the analysis is actually barrier-aware.**
  If a cross-barrier tract is excluded or downweighted by the approved method,
  it simply is not filled, or is filled at the approved lower weight.
- **No drive-time isochrones** unless a separate owner-approved design explains
  how a static site will precompute and maintain them.

## Hard Rules

- No fabricated calibration factors.
- No arbitrary AADT, river-size, rail, or commute-flow thresholds without cited
  calibration evidence.
- No synthetic LODES fallback in shipped analysis.
- No new scoring movement without a migration table in `docs/PMA_SCORING.md`
  and re-pinned regression fixtures.
- No decorative geometry trimming.
- No production code before the RFC answer is reviewed.
- Loopback fixture URLs only.
- External QA gates every PR.

## RFC Deliverable

Create a new scoping document or issue comment that answers every item below.
This RFC may include prototype notebooks/scripts for measurement, but those
must be clearly marked non-production and must not change public behavior.

### 1. Barrier-Aware Method Options

Compare at least two implementable methods:

- **Tract graph connectivity:** build tract adjacency from the simplified tract
  geometry, then remove/downweight adjacency edges crossed by hard barriers.
- **Site-to-tract crossing screen:** test whether the site-to-tract centroid or
  site-to-tract representative point crosses barrier features.

For each method, document:

- required input files,
- computational cost in browser and/or build step,
- known false positives and false negatives,
- how it handles bridges, mountain highways, rivers, reservoirs, rail, and
  tract boundaries that already follow barriers,
- whether it excludes tracts or applies fractional downweights,
- what public-facing disclosure would say.

### 2. Barrier Feature Taxonomy

Audit `data/market/natural_barriers_co.geojson` before using it. The RFC must
summarize counts by `barrier_type` and any available subtypes such as route
class, AADT, water geometry type, name, or source layer.

The RFC must propose an explicit taxonomy:

- **Hard barrier candidate:** likely to prevent practical local market access
  unless a crossing exists.
- **Soft/friction barrier candidate:** may reduce practical access but should
  not exclude a tract by itself.
- **Context-only feature:** useful map context but not an analytic input.

Do not inherit the existing `MIN_BARRIER_AADT = 10000` or exclusion percentages
from `js/pma-barriers.js` without calibration.

### 3. Crossing and Connectivity Logic

The RFC must specify how crossings are recognized. At minimum, address:

- highway bridges/interchanges and river crossings,
- water bodies crossed by highways or local roads,
- tracts connected only by a long detour,
- cross-barrier tracts whose centroid is near the site but whose neighborhood
  access is practically separated,
- large rural/resort tracts where centroid logic is especially fragile.

If crossings cannot be represented with current committed data, the RFC must
say so and keep the affected method out of scope.

### 4. Calibration Evidence

Before implementation, compare the proposed barrier-aware output against
**one to two professional market studies** or equivalent owner-approved
benchmarks. The benchmarks must be fetchable or provided as review artifacts.

For each benchmark, include:

- site or approximate location,
- professional PMA description or map,
- current tract-buffer PMA result,
- proposed barrier-aware result,
- what changed and whether the change is directionally correct,
- unresolved mismatch notes.

If no benchmark is available, stop. Do not invent calibration.

### 5. Commute-Shed Extension

The RFC must separately scope commute-shed behavior. It is not automatically
part of barrier-aware PMA.

Required decisions:

- Is commute-shed a scored analytic extension, a context overlay, or both?
- Does it add tracts beyond the circular buffer, or only annotate existing
  included tracts?
- Which LODES vintage and field set is authoritative?
- How are workplace tracts, home-origin tracts, and likely tenant-origin tracts
  distinguished?
- What capture threshold, if any, is defensible?
- How does commute-shed interact with hard barriers?

Hard guard: `js/pma-commuting.js` currently contains synthetic fallback data
and convex-hull boundary generation. Those paths must not ship as analysis.

### 6. User Experience and Disclosure

The RFC must define the public UI contract before implementation:

- labels for circular-buffer PMA, barrier-aware PMA, commute-shed overlay, and
  any context-only layers,
- whether the user can toggle barrier-aware and commute-shed modes,
- how score movement is disclosed,
- what appears in exports or saved PMA summaries,
- how unavailable or partial data is surfaced.

Keep the current whole-tract opacity rule. Do not introduce clipped tract arcs.

### 7. Test and QA Plan

The RFC must propose non-vacuous tests for any implementation PR:

- fixture where a known cross-barrier tract is excluded/downweighted,
- fixture where a nearby same-side tract remains included,
- fixture where removing the barrier file disables barrier behavior and surfaces
  a warning instead of silently changing scores,
- fixture proving synthetic commute fallback cannot affect output,
- fixture proving rendered tract set matches the analytic tract set,
- score migration table for urban, suburban, rural, and resort sites if scores
  move.

External QA will sabotage the barrier logic by disabling one crossing/exclusion
branch and expect at least one test to fail.

## Recommended Sequencing After RFC Approval

The RFC should recommend PR splits, but this is the safest default:

1. **PR C1 — data/method audit only:** no public behavior change. Add source
   inventory, barrier taxonomy, calibration fixtures, and tests that prove the
   source files are non-vacuous.
2. **PR C2 — barrier-aware prototype behind an explicit disabled flag:** no
   default score change. Validate tract-set changes against approved fixtures.
3. **PR C3 — enable barrier-aware mode if approved:** disclose score movement,
   update `docs/PMA_SCORING.md`, re-pin PMA scoring fixtures, and add browser
   QA evidence.
4. **PR D — commute-shed, if approved:** separate from barriers unless the RFC
   proves the two must ship together.
